# SDD执行总览

## 1. 目标与范围

本文件定义 Stratum 第一阶段（v1.0）规范驱动开发流程，覆盖：
- 架构约束
- 编码前置规范
- 验收标准
- 变更控制

本阶段不包含：微服务拆分、多区域容灾。

## 2. SDD 工作流（强制）

1. 需求规范化
2. 架构与接口规范确认
3. 安全与数据权限规范确认
4. 可观测性与运维规范确认
5. 任务拆解与实现
6. 自动化校验与人工验收

规则：未完成上一阶段验收，不得进入下一阶段。

## 5. Definition of Done（统一完成标准）

一个需求项完成必须同时满足：
- 文档：对应规范已更新且无冲突
- 代码：实现通过静态检查与单元测试
- 接口：OpenAPI 与响应模型符合 API 规范
- 文档：OpenAPI JSON 与 Knife4j UI 入口可访问且内容一致
- 查询：Query 基架接口契约与分页排序规范通过
- 安全：通过鉴权、授权、数据权限回归用例
- 观测：具备 traceId、结构化日志、核心指标
- 任务：后台任务具备幂等、并发保护与失败重试
- 消息：消息总线具备 Outbox、重试、死信与可追踪性
- 发布：可通过 compose 启动依赖并完成本地验收

## 6. 变更控制

变更分级：
- A 类（破坏性）：需要评审并更新版本说明
- B 类（兼容性增强）：更新对应专题文档
- C 类（文案修正）：直接修订并记录

流程：
1. 提交变更提案（影响面、风险、回滚）
2. 更新规范文档
3. 实施与测试
4. 记录结果（验收证据）

## 7. 冲突处理规则

- 同一主题只允许一个权威来源。
- 检测到冲突时，先修正文档再允许合并代码。
- 无法当场判定时，按“更严格约束优先”执行。

## 8. 执行产物

每个迭代必须输出：
- 规范变更记录
- Spec 清单与任务映射
- AI 可执行开发节点计划
- 可追溯任务清单
- 测试与验收记录
- 部署与回滚说明
