# 安全认证设计

## 1. 安全目标

- 无状态认证
- 精细授权
- 数据权限控制
- 会话可撤销
- 全链路可审计

## 2. 认证机制

### 2.1 Token 体系

- Access Token：有效期 15 分钟
- Refresh Token：有效期 14 天
- Refresh 必须旋转，旧 token 立即失效

### 2.2 签名与字段

- 算法：RS256
- 必含 claims：sub、iat、exp、jti、sid、ver
- 可含 claims：tenantId、orgId、deptId

## 3. 撤销机制（强制）

### 3.1 权威存储

- Refresh 白名单：auth:rt:{jti}
- 会话状态：auth:sess:{sid}
- 用户版本：auth:user:{userId}:ver

### 3.2 立即失效规则

- 改密、封禁、强制下线时必须递增 user ver
- Access 校验必须比较 jwt.ver 与 Redis ver

## 4. RBAC 与数据权限

### 4.1 RBAC 模型

- user、role、permission、user_role、role_permission
- permission.type 仅允许 API 与 FEAT 两类

### 4.2 数据范围模型

- ALL、ORG、ORG_AND_SUB、DEPT、DEPT_AND_SUB、SELF、CUSTOM
- 查询必须附加数据范围过滤
- 写操作必须做目标对象范围校验

## 5. 前端功能权限

- 后端通过 auth/me 下发 featurePermissions
- 前端仅用于展示控制
- 后端 API 权限校验不可省略

## 6. 统一安全事件

- AUTH_LOGIN_SUCCESS
- AUTH_LOGIN_FAIL
- AUTH_REFRESH_SUCCESS
- AUTH_REFRESH_FAIL
- AUTH_LOGOUT
- AUTH_KICK
- AUTH_TOKEN_REVOKE
- AUTH_FORBIDDEN
- AUTH_DATA_FORBIDDEN

## 7. 接口约束

- POST /auth/login
- POST /auth/refresh
- POST /auth/logout
- POST /auth/logout-all
- GET /auth/me

所有接口必须返回 traceId。

## 8. 可执行验收

- Access/Refresh 生命周期符合规范
- logout-all 可使历史 access 立即失效
- RBAC 与数据权限用例覆盖通过
- 403 与 401 错误码语义准确
- 安全事件完整入库与入日志
